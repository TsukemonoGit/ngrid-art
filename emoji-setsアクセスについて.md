emoji-sets ページの直接アクセス分析

### ページの役割

このページは Nostrリレー上にあるすべての kind:30030（絵文字セット）をフェッチし、グリッド表示する公開ページ 。

### 実行フロー

```
  ページ読み込み
    → onMount()
      → waitForRelayReady() → リレー接続を待つ
      → latestEmojisFromOthers.size === 0 なら fetchMore()
        → fetchAllKind30030FromOthers() → filter.until = 現在のタイムスタンプ
        → Nostrリレーから過去200件(=FETCHLIMIT)の kind:30030 を取得
      → $effect → 登場pubkeyの kind:0 (プロフィール) をフェッチ
      → kind30030Events が computed される
         ├→ latestEmojisFromOthers.value (他人の最新30030)
         └→ kind30030Stock.value (登録済み30030)
            をマージ・ソートして表示用配列に
```

### 具体的に何が起きるか

1.  ページは表示される — 認証ガードがない
2.  他人の絵文字セットが読み込まれる — Nostrリレーからkind:30030をフェッチ
3.  「追加」「削除」ボタンは表示されない — loginUser.value が falsy なので EmojiSetCard 内で {#if loginUser.value} が false
4.  プロフィール画像/表示名はフェッチされる — $effect で fetchKind0ForPubkeys() が走る（ただしkind0キャッシュが空なら表示されるまで少し時間がかかる）
5.  「もっと読む」で追加フェッチ可能 — ページネーションは動作する  


### 潜在的な問題

- エラーハンドリングがない: waitForRelayReady() や fetchAllKind30030FromOthers()  
  のrejectをcatchしていない。リレー接続が失敗すると「nodata」が表示されるが、ユーザーにはなぜわからない
- kind30030Eventsの計算で?.が不足: (kind10030.value?.tags ?? []) は安全だが、latestEmojisFromOthers.value や kind30030Stock.value は空Mapが初期値なので問題なし。ただし  
  loginUser.value が '' であることは、後でログインした際に再フェッチなどのトリガーにならない（明示的なログイン検知が必要）
